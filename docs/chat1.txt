  Scope

  Two parallel build tracks:
  - Track A ‚ÄĒ order-service/: NestJS 10 scaffold from scratch. No domain

  Key adaptations from the references

  - serviceBv2 and serviceBv2Core use raw Express + TSyringe. We don't copy
  that. We adapt their infra logic (Rabbit client, sharded Knex resolver, region
   resolver semantics, core-events consumer flow, WS handshake, outbox drain)
  into NestJS-native shapes (modules, providers, @Injectable, OnModuleInit /
  OnApplicationShutdown) following core-service/backend/'s exact conventions.
  - pkg/ files copied largely verbatim (they're already framework-agnostic).
  - Where serviceBv2's implementation uses an AppError class, we use NestJS's
  BadRequestException/UnauthorizedException/ServiceUnavailableException etc.,
  per CLAUDE.md ¬ß10.
  - Region resolver: per CLAUDE.md ¬ß8 it's middleware, per implementation-plan
  ¬ß0.3 it reads X-Region only (no path/query/cookie/JWT fallback). I'll go with
  the plan's stricter rule.
  - Idempotency: copied from core verbatim ‚ÄĒ DB-fallback hook stays disabled
  until Phase 1 lands the idempotency_keys table.
  - Logging: copied from core. The logs table migration moves into order-service
   migrations even though no other Phase 0 migration is created (CLAUDE.md and
  the implementation-plan ¬ß0.4 say "no migrations in Phase 0"). Decision: I'll
  skip the logs-table migration in Phase 0 to honor ¬ß0.4 strictly, and disable
  the DB write path (log to console only) until Phase 1 ships a logs migration.
  Will flag this for your call.
  - WS: @nestjs/websockets + @nestjs/platform-socket.io +
  @socket.io/redis-adapter, gateway with afterInit adapter install,
  handleConnection JWT verify + room joins, WsPublisher provider. Token via
  auth.token or cookie (per ref) ‚ÄĒ query-string omitted because of log leakage;
  doc-string says cookie/?token=, but ref's reasoning is correct. I'll match the
   ref (cookie + auth.token).
  - Health: NestJS @nestjs/terminus plus a pingAll() shard fan-out (uses
  KNEX_CONNECTION provider's pingAll).
  - Core-client base: NestJS @Injectable provider exposing a request<T>()
  method. Retry via pkg/utils/retry.ts. Forwards X-CorrelationId from
  RequestContextService. Translates non-2xx to NestJS HttpException subclasses
  (no custom AppError).
  - AMQP module: single shared RabbitMQClient connection (auto-reconnect via
  amqp-connection-manager), opened in OnModuleInit, closed in
  OnApplicationShutdown.
  - Core-events consumer: Nest module whose OnModuleInit declares topology,
  registers consumer, dispatches via a handler registry. Registry exposes a
  register(eventType, handler) API for later phases. Phase 0 starts empty.
  - DI strategy: Knex provider stays under the same KNEX_CONNECTION token
  (matches core), but the value is { db(region), dbArchive(region), pingAll() }
  ‚ÄĒ the sharded resolver (CLAUDE.md ¬ß8). Repos in later phases will inject and
  call this.knex.db(region) rather than the bare Knex.
