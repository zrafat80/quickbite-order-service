  Phase 3 (compressed) — Deliveries + Agents in one shot

  Schema changes (no deliveries table — per PNG)

  The lifecycle assigned → picked → delivered already lives on orders.status
  with timestamps (assigned_at, picked_at, delivered_at), and
  orders.delivery_agent_id already names the current agent. The only thing the
  dropped deliveries table was carrying that the order itself doesn't is the
  reassignment chain + a per-attempt audit. We replace those with:

  1. Modify 20260507000020_create_orders.ts (in place — drop existing rows, no
  new migration):
  - Add assignment_attempts INT NOT NULL DEFAULT 0 — bumped each time we hand
  the order to an agent; enforces MAX_REASSIGNMENT_ATTEMPTS.
  - Add last_assignment_at TIMESTAMP(3) NULL — for the assignment-loop watchdog.
  - Keep commission INT NOT NULL DEFAULT 0 as-is. Note: commission
  computation/insertion is deferred to Phase 5 (finance). Phase 3 leaves it 0.
  - Audit of who rejected lives in Redis (assign:reject:<orderId> set, TTL =
  lifetime of the assignment loop) — ephemeral, no SQL needed.

  2. New migration …_create_agent_presence.ts — exactly as in database-design.md
   §3.8. PostGIS extension, single row per agent, partial GIST index WHERE
  is_online=TRUE, partial btree on last_seen_at for the stale-cleanup scan.

  3. New migration …_create_agent_earnings.ts — same as §3.9, but unique
  constraint moves from delivery_id to order_id (since no delivery row). FK is
  just to orders(id, created_at) (composite, partitioned-table rule). One
  earning row per order ⇒ insert is idempotent in the delivered trx.

  Redis keys (hot path)

  Key: presence:geo:<region>
  Type: GEO zset
  Purpose: online agents' coordinates
  ────────────────────────────────────────
  Key: presence:meta:<region>:<agentId>
  Type: hash
  Purpose: last_seen_at, active_orders, is_online
  ────────────────────────────────────────
  Key: presence:busy:<region>
  Type: set
  Purpose: derivable from active_orders>0; kept for fast filtering
  ────────────────────────────────────────
  Key: assign:reject:<orderId>
  Type: set + TTL
  Purpose: agents who rejected/timed-out for this attempt

  Smart Assignment Loop (your spec, with one cleanup)

  1. Branch coords from BranchClient (cached via core-events invalidation)
  2. GEOSEARCH presence:geo:<region> BYRADIUS <branch.lng,branch.lat> 5000m
  WITHCOORD ASC COUNT 5
  3. PIPELINE: HMGET presence:meta:<region>:<agentId> last_seen_at active_orders
    (×5)
  4. Drop: now - last_seen_at > 300s   (5-min strict cutoff per your call)
          OR agent ∈ assign:reject:<orderId>
          AND lazily SREM/ZREM the stale ones from Redis
  5. Sort: (active_orders ASC, distance ASC)
  6. For array[0]:
       - trx: SELECT agent_presence FOR UPDATE; verify is_online & last_seen >
  NOW()-5min
       - UPDATE orders SET status='assigned', delivery_agent_id=?,
  assigned_at=NOW(),
                            assignment_attempts = assignment_attempts + 1,
                            last_assignment_at = NOW()
         WHERE id=? AND status='ready'
       - HINCRBY presence:meta active_orders +1; SADD presence:busy
       - commit
       - WS task.assigned → agent:<id>; await accept within
  AGENT_ACCEPT_TIMEOUT_SEC (30s)
  7. On reject/timeout: SADD assign:reject:<orderId>; HINCRBY active_orders -1;
                       re-flip orders.status to 'ready'; pick array[1]; repeat
  8. After MAX_REASSIGNMENT_ATTEMPTS: WS admin alert; leave order 'ready'

  PRESENCE_STALE_SEC default goes from 90 → 300 to match your 5-minute rule.

  Settlement on delivered (Phase-3 scope only)

  In one trx with the status flip:
  1. UPDATE orders.status='delivered', delivered_at=NOW().
  2. For COD: flip transactions(type='cod_collection') from pending → succeeded.
  3. INSERT agent_earnings(agent_id, order_id, amount, currency) — unique on
  order_id makes it safe.
  4. HINCRBY presence:meta active_orders -1; SREM presence:busy if zero; SREM
  assign:reject:<orderId>.

  Deferred to Phase 5 (finance): commission compute,
  transactions(type='commission') insert, restaurant_balances.balance +=
  subtotal - commission. Phase 3 leaves balance alone.

  Endpoints landed in this phase

  - Deliveries (acting on the order, no /deliveries/:id): POST
  /orders/:id/assign (admin manual), POST /orders/:id/reassign, PATCH
  /orders/:id/delivery-status with {accept|pickup|deliver|reject} (agent-only,
  JWT-asserted owner of orders.delivery_agent_id).
  - Agents: POST /agents/presence/{online|offline|ping}, GET
  /agents/tasks?status=, GET /agents/earnings?from&to.

  Auto-assignment on ready is invoked in-process from OrderService.updateStatus
  after commit (same hook the original Phase-3 plan had).

  One alternative worth flagging on auto-assignment

  Your loop is fine; the only race it doesn't cover is two orders dispatching
  simultaneously picking the same array[0] agent. Two ways to harden it later if
   it shows up:
  - Cheap: rely on the SELECT agent_presence FOR UPDATE + post-commit HINCRBY —
  losers just retry array[1]. Good enough for v1.
  - Stronger: a small Lua script that pops the best candidate atomically
  (GEOSEARCH + HMGET + SADD claim in one round-trip). Worth doing once we see
  contention; not for now.

  I'd start with the cheap version.
